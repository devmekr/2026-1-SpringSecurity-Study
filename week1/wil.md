## 1. DTO를 Record로 설계하는 이유

DTO는 데이터 전달을 위한 객체이기 때문에 상태를 변경하는 로직보다는 단순히 값을 담는 역할에 집중하는 것이 중요하다. 기존 클래스 방식으로 DTO를 작성하면 생성자, getter, equals, hashCode 등을 반복적으로 작성해야 하지만, record를 사용하면 이러한 보일러플레이트 코드를 줄일 수 있다.

또한 record는 기본적으로 불변(immutable) 객체이기 때문에 의도치 않은 값 변경을 방지할 수 있다. 이는 여러 계층에서 DTO를 사용할 때 발생할 수 있는 사이드 이펙트를 줄여주며, 안정적인 데이터 전달을 가능하게 한다. 따라서 DTO는 record로 설계하는 것이 코드의 간결성과 안정성 측면에서 유리하다.

## 2. of / from 메서드 패턴

객체를 생성할 때 단순히 생성자를 사용하는 대신, 정적 메서드를 활용하면 생성 의도를 더 명확하게 드러낼 수 있다.

* of: 값을 직접 받아 객체를 생성하는 경우 사용
* from: 기존 객체를 변환하여 새로운 객체를 생성하는 경우 사용

예를 들어, DTO를 생성할 때 단순 값으로 생성한다면 of를 사용하고, Entity를 DTO로 변환하는 경우에는 from을 사용하는 것이 자연스럽다. 이러한 방식은 코드 가독성을 높이고, 객체 생성 과정의 의미를 명확하게 전달해준다.

## 3. 인증(Authentication)과 인가(Authorization)

인증은 사용자의 신원을 확인하는 과정으로, 흔히 로그인 과정과 연결된다. 즉, “너 누구야?”에 대한 질문에 해당한다. 반면 인가는 인증된 사용자가 특정 자원에 접근할 수 있는 권한이 있는지를 확인하는 과정으로, “너 여기 들어갈 수 있어?”에 해당한다.

예를 들어, 로그인에 성공하면 인증이 완료된 것이고, 이후 특정 페이지에 접근할 수 있는지 여부는 인가 과정에서 판단된다.

## 4. Spring Security 개요

Spring Security는 Spring 환경에서 인증과 인가를 담당하는 프레임워크이다. 사용자의 신원을 확인하고, 적절한 권한을 부여하여 리소스 접근을 제어하는 역할을 한다.

강의에서는 이를 건물 보안 시스템에 비유하였다. 인증은 신분증을 확인하고 출입증을 발급하는 과정이며, 인가는 해당 출입증을 기반으로 특정 공간에 접근할 수 있는지를 판단하는 과정이다.

## 5. Filter와 Spring Security 동작 방식

Spring Security는 서블릿 필터 기반으로 동작한다. 서블릿 필터는 요청과 응답을 가로채어 전처리 및 후처리를 수행하는 역할을 한다.

예를 들어 요청 로깅, 인코딩 설정과 같은 작업이 필터에서 이루어진다. Spring Security 역시 이러한 필터 구조를 활용하여 보안 처리를 수행하며, 이를 SecurityFilterChain이라고 한다.

즉, 클라이언트의 요청이 컨트롤러에 도달하기 전에 여러 보안 필터를 거치게 되며, 이 과정에서 인증과 인가가 처리된다.

## 6. 인증 및 인가 흐름

인증 과정에서는 사용자가 로그인 요청을 보내면 인증 필터가 이를 가로채고, AuthenticationManager를 통해 인증을 수행한다. 이후 UserDetailsService를 통해 실제 사용자 정보를 조회하고 인증 결과를 반환한다.

인가 과정에서는 요청한 경로에 대해 사용자가 접근 가능한 권한을 가지고 있는지 판단한다. 예를 들어, 특정 경로는 ADMIN만 접근 가능하고, 어떤 경로는 로그인한 사용자만 접근 가능하며, 나머지는 누구나 접근 가능하도록 설정할 수 있다.
